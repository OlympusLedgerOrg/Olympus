## 1. Executive summary

The frontend is in very good shape for a single-origin Tauri desktop
webview. Highlights:

- **No XSS surfaces.** Zero uses of `dangerouslySetInnerHTML`, zero raw
  HTML interpolation, every value reaches the DOM through JSX text nodes.
- **No `eval` / `new Function` / inline `<script>`.** CSP-compatible.
- **In-memory secret handling.** API key and admin key live in module-scoped
  variables in `src/lib/storage.ts`; never written to localStorage,
  sessionStorage, or IndexedDB.
- **External links carry `rel="noopener noreferrer"`.**
- **BLAKE3 WASM has an all-zero output guard** to catch ABI mismatches before
  a corrupted hash silently desynchronises from the Rust hot path.

Five findings, all minor: **0 High, 2 Medium, 3 Low**.

## 2. Findings

### 🟡 M-UI-1 — `useFileCommit` surfaces backend `detail` verbatim on 401

**Location:** [hooks/useFileCommit.ts:117](app/public-ui/src/hooks/useFileCommit.ts:117).

**What:** The browser-fallback path in the file-commit hook interpolates
`detail` (the backend's 401 response body) directly into the user-visible
error string:

```ts
res.status === 401
  ? `Authentication failed (${detail}) — paste a valid API key…`
  : detail
```

The auth middleware's 401 body can carry context that we'd rather not put on
screen (key prefix, expiry/revocation reason, fingerprint). Other 4xx codes
also surface `detail` verbatim, but 401 is the only one we can safely
generalise without losing UX value.

**Fix:** Swallow `detail` on 401 and show a fixed message:
```ts
if (res.status === 401) {
  throw new Error("Authentication failed — paste a valid API key in the box above and try again.");
}
const detail = …; throw new Error(detail);
```

### 🟡 M-UI-2 — Clipboard secret-copy buttons never auto-clear

**Location:** `InitialSecretsModal.tsx`, `StartupGate.tsx` (and any other
"Copy" button that writes a secret via `navigator.clipboard.writeText`).

**What:** Once the user clicks "Copy", the API key / BJJ private key stays
on the clipboard indefinitely (until the user copies something else). Any
clipboard-sync layer (cloud-sync clipboard managers, OS-level history,
foreground app that polls clipboard) can read it.

**Fix sketch (Claude Design owns the UX):**
- After successful `writeText`, queue a `setTimeout` (~30 s) that overwrites
  the clipboard with an innocuous value, OR
- Use the OS-level "secret clipboard" API where available (none today on
  Windows / Linux, partial on macOS), OR
- Surface a clear UX affordance ("Copied — clipboard will clear in 30 s").

**Flagged, not fixed:** UI behaviour change — Claude Design's territory.

### 🟢 L-UI-1 — Vite production build ships sourcemaps by default

**Location:** [vite.config.ts](app/public-ui/vite.config.ts).

**What:** No `build.sourcemap` field, so Vite emits `.map` files into
`dist/` on `pnpm build`. The Tauri resource glob then bundles those into
the installer. A user with read access to the install directory can
reconstruct the original TypeScript source, including the key-handling
code paths and storage helper logic.

**Fix:** `build: { sourcemap: false }` in `vite.config.ts`.

### 🟢 L-UI-2 — React Query cache keys don't include an API-key fingerprint

**Location:** All `useQuery` calls in `app/public-ui/src/pages/*.tsx`.

**What:** The audit raised the possibility of cross-key cache pollution if
the user switches API keys mid-session. **Reviewing the current query
inventory shows no actual collision**: the only cached queries are
`public-stats`, `record-proof`, `dataset-detail`, and `dataset-verify` —
all of which are either public/unauthenticated or scoped by a deterministic
record/dataset id, not by API key.

**Why kept as a finding:** The pattern is fragile. Any future addition of
an API-key-scoped query (e.g. `["my-credentials"]` listing the holder's
SBTs) would silently inherit the cross-session leak risk.

**Fix:** Document the convention — protected queries must include an
`apiKeyFingerprint` (BLAKE3-truncated) element in the `queryKey` tuple.
Defer the structural change until the first such query is added.

### 🟢 L-UI-3 — Backend error `detail` displayed verbatim throughout the UI

**Location:** [lib/api.ts:131](app/public-ui/src/lib/api.ts:131)
(`apiFetch`'s `ApiError` carries `detail`); consumers surface `err.message`.

**What:** Defense in depth — if the backend ever regresses on its
"generic error message" hardening (audit TOB-OLY-07), the frontend will
faithfully surface any leaked internals.

**Why not fixed now:** The backend already returns generic strings
("Database error", "Server error", etc.) via the `db_err` and shared error
helpers in every audited Axum module. Adding a frontend whitelist on top
would either duplicate that policy (drift risk) or strip useful information
(e.g. the `required_scope` field).

**Fix sketch (deferred):** If the backend audit posture ever softens,
introduce `userFacingError(err: unknown): string` in `lib/api.ts` and
require all error UIs to route through it.

### 🟢 L-UI-4 — Client-side route gating could be mistaken for security

**Location:** [components/Layout.tsx:23](app/public-ui/src/components/Layout.tsx:23)
(`canManageKeys = hasStoredAdminKey()`).

**What:** The nav bar hides KEYS / USERS / SBTs entries when no admin key
is present. This is correct (advisory UX), but the line has no comment
explaining that it is **not** an authoritative gate. A future contributor
adding a new admin-scoped feature could reach for `canManageKeys` as the
sole protection and skip the backend scope check.

**Fix:** Add a comment at the `canManageKeys` declaration explaining the
boundary: backend enforces scope; this flag is UX-only.

## 3. Verified clean — appendix

| Item | Why it's clean |
|---|---|
| `dangerouslySetInnerHTML` | Zero occurrences across the codebase. |
| `eval` / `new Function` / inline `<script>` | Zero occurrences; CSP-compatible. |
| `target="_blank"` external links | Both external links carry `rel="noopener noreferrer"`. |
| `postMessage` listeners | None registered. The app does not consume cross-origin messages. |
| `navigator.clipboard.readText` | Only used in flows where the user has just clicked a "paste API key" affordance. |
| `localStorage` / `sessionStorage` of secrets | `storage.ts` keeps API key + admin key in module-scoped variables. The only `localStorage` writes are skin/preference state. |
| API key sent as `X-API-Key` only | `lib/api.ts` headers; never URL query, never request body. |
| BLAKE3 WASM ABI guard | `blake3.ts` rejects all-zero output to catch incorrect WASM loader paths before a corrupted hash desynchronises from Rust. |
| File drag-drop | Uses native `<input type="file">` + Tauri picker; no host-FS traversal. |
| Telemetry / Sentry / GA | Not present. No external network calls outside the embedded Axum server. |
| Tauri IPC gating | All `invoke()` calls guarded behind `__TAURI_INTERNALS__` checks; falls back to direct fetch in the dev browser path. |
| React Query mutation invalidation | `useMutation` callsites call `queryClient.invalidateQueries` or local refetch helpers; no silent staleness. |
| Startup unlock | PBKDF2 via `crypto.subtle` with 160k iterations — sufficient for the desktop threat model. |
