# Audit — Tauri 2 IPC command surface

**Date:** 2026-05-25
**Branch:** `claude/tauri-ipc-hardening` (HEAD `2df0bffe`)
**Scope:** Every `#[tauri::command]` in `src-tauri/`, the
`tauri::generate_handler![...]` registration in `main.rs`, `tauri.conf.json`
(capabilities / CSP / allowlist / plugin registrations), and
`tauri_plugin_*` use. Excludes the Axum HTTP surface (separate
2026-05-25 audit) and ZK / anchoring / federation crypto (also separately
audited).
**Type:** Read-only audit.

---

## 1. Executive summary

The IPC surface is **6 registered commands** plus **1 unregistered
dead-code command** (`zk::commands::verify_proof`). The branch's recent
hardening commits — `66e117a0` (Vec<u8> cap + zeroize), `a4afd76a` (TOCTOU
in `open_file_dialog`), `17cd15ea` (`is_file()` guard + dropped derive
feature), `2df0bffe` (drop `Clone` from secrets struct) — close the
serious holes (F-2, F-4, file-pick TOCTOU).

Three findings remain, all minor: **0 High, 1 Medium, 2 Low (one of which
is a dup of the medium's adjacent doc issue)**.

The headline observation is **not** a vulnerability — it's a maintenance
gap: a `#[tauri::command]` function exists with a docstring that says
"called from the frontend via `invoke('verify_proof', …)`", but the
command is **not** registered in `generate_handler![]`. The function is
unreachable today and the docstring is actively misleading; left as-is, a
future contributor could wire it into the handler (closing the dead-code
gap) without re-examining input validation appropriate for IPC exposure.

## 2. Findings

### 🟡 M-IPC-1 — `commit_file` accepts unbounded `file_name` String

**Location:** [main.rs:47](src-tauri/src/main.rs:47) (`commit_file`).

**What:** `file_bytes` is capped at `IPC_BYTES_LIMIT` (128 MiB, audit F-2),
`shard_id` is sanitised at the Axum boundary by `sanitize_shard`, and
`record_id` is capped at 256 chars and control-char-checked. The
`file_name` argument has **no length cap** before it is handed to
`reqwest::multipart::Part::bytes(...).file_name(file_name)`.

A misbehaving (or compromised) webview can submit a multi-megabyte
filename. The multipart encoder will write it into the Content-Disposition
header, log lines, and any error path that includes the filename — none of
which are bounded today.

**Why it matters:** Defense in depth at the IPC boundary. Even if no
downstream path crashes on a long filename, the cost of validating is one
line and the cost of leaving it open is unbounded memory in log/error
strings.

**Fix sketch:** Add a 256-byte cap matching `record_id`'s shape, returning
the same Err-formatted message used for `file_bytes` overflow.

### 🟢 L-IPC-1 — Unregistered `verify_proof` IPC command + misleading docstring

**Location:** [zk/commands.rs:65](src-tauri/src/zk/commands.rs:65),
[main.rs:567](src-tauri/src/main.rs:567).

**What:** `zk::commands::verify_proof` is decorated with
`#[tauri::command]` and its docstring states:
> Called from the frontend via `invoke("verify_proof", { circuit, proofJson, publicSignals })`.

But the `generate_handler![...]` block in `main.rs:567` lists only:
`get_api_port`, `get_db_error`, `commit_file`, `take_initial_secrets`,
`get_startup_error`, `open_file_dialog`. `verify_proof` is **not**
registered and is unreachable from the webview.

The frontend uses the Axum route `POST /zk/verify` (with auth + scope
gates) for the same purpose, which is the right design — the IPC version
would have to repeat those checks and currently does not.

**Why it matters:** Dead code with a misleading docstring is one
short-attention-span PR away from being live IPC without the validation
that the HTTP route applies (auth, rate limiting, scope check). Either
delete it or register it deliberately and add the gates.

**Fix sketch:** Delete `src-tauri/src/zk/commands.rs` and remove the
`pub mod commands;` + `pub use commands::verify_proof;` lines from
`zk/mod.rs`. The HTTP route already covers the use case.

### 🟢 L-IPC-2 — `zk::commands` re-exported despite being unused

**Location:** [zk/mod.rs:6](src-tauri/src/zk/mod.rs:6),
[zk/mod.rs:18](src-tauri/src/zk/mod.rs:18).

**What:** `zk/mod.rs` does `pub mod commands;` and `pub use
commands::verify_proof;`. Neither is imported anywhere outside the
module (the only callers of `verify_proof` in the workspace are unit
tests using the `ark_groth16::Groth16::verify_proof` static method, a
different function with the same name).

**Fix sketch:** drops along with L-IPC-1.
